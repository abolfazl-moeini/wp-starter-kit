01 — Why

Why WPDev Framework

A clean split between the framework and the optional domain layer. No marketing. No “it just works”. The architecture is the message.

modules/

The framework layer

Everything in modules/ is required. It is the spine: module loading, lifecycle helpers, service registry, builders, dashboard/settings/wizard shell. Removing a module folder may fatal.

  • 13 modules with setup.php, API_DOC.md, README.md
  • Uniform wpdev_register_* registry facades
  • Autoloader + Legacy_Shim + Examples_Shim compatibility
wpdev-examples/

The optional WaaS layer

Domain code, playground demos, and sample panels live under wpdev-examples (sibling plugin). It is deletion-safe by contract: removing an example should only remove that example’s admin UI/domain behavior.

  • 21 documented API examples (products, customers, sites, gateways, …)
  • Folder slug is the bare name: wpdev-examples/products/, not wpdev-products/
  • New domains belong here — never under inc/
inc/

Legacy artifact only

inc/ holds the legacy monolith artifact. It must stay PHP-free. The old modules/wpdev-* paths are also gone — domain code is now under wpdev-examples (sibling plugin).

  • Do not add PHP back under inc/
  • Do not create modules/wpdev-* domain modules
  • Do not reference old monolith paths from new code
Goal

Built for WordPress admin

WPDev is a framework for building WordPress admin panels, CRUD screens, settings pages, dashboards, forms, widgets, playground panels, and optional WaaS integrations. It is multisite-first.

  • Reusable base classes, builders, and facades
  • Production-ready dashboard, settings, wizard shells
  • Designed to be extended, not configured
02 — Architecture

Entry points, lifecycle, module loading

Boot is deterministic. Modules are discovered, registered, and instantiated on well-known hooks. No surprises.

Entry points

  1. WordPress loads wpdev.php
  2. wpdev.php defines WPDEV_PLUGIN_FILE and requires constants.php, autoload.php, and the dependency bootstrap
  3. It boots the dependency manager via wpdev_bootstrap_dependency_manager( __DIR__ )
  4. Core autoloaders (Module_Autoloader, Legacy_Shim_Autoloader, Examples_Shim_Autoloader, root Autoloader) are wired
  5. Action Scheduler is loaded from modules/core/dependencies/
  6. The singleton trait and WPDevFramework\Core\WPDev are required
  7. The wpdev() helper is defined
  8. On plugins_loaded, $GLOBALS['WPDev'] = wpdev()
  9. modules/core/setup.php is required, which registers the core module
  10. wpdev-examples/wpdev-examples.php loads optional WaaS modules is loaded as the backward-compatible module-loader entry

Core setup

  • modules/core/setup.php registers the core module
  • On plugins_loaded priority 5, Module_Loader::load_all() discovers modules/*/setup.php
  • On wpdev_modules_loaded priority 10, wpdev_load_examples() loads optional examples
  • On wpdev_init priority 0, Service_Registry::boot() boots services
  • When the wpdev-playground plugin is active (WPDEV_PLAYGROUND_DIR), it boots playground infrastructure on plugins_loaded priority 4

Registry facade pattern

Every builder module exposes a uniform registry. This is the contract you should reach for first. Examples override the entity name (form, field_type, settings_section, …). 

wpdev_register_{entity}( $id, array $config = array(), bool $replace = true ): bool
wpdev_get_{entity}( $id )
wpdev_has_{entity}( $id ): bool
wpdev_list_{entity}(): array
wpdev_unregister_{entity}( $id ): void

Metaboxes are scoped per page: wpdev_register_metabox( $page_id, $metabox_id, $config ).

01

core

Dependencies: none · Foundation module

← back to grid

The foundation. Owns module loading, lifecycle helpers, service access, admin-page rebinding, playground infrastructure, events, public function maps, and compatibility bootstrapping.

When to use

  • Load another framework module with dependencies
  • Register callbacks on the WPDev lifecycle
  • Access framework services such as ajax, form, modal, screen options, tour, or view
  • Rebind a production admin page under another menu / context
  • Register playground panels
  • Dispatch or listen to framework events

Key APIs

  • wpdev_load_module( $module_id, $modules_dir = '' ) — load one module and its dependencies
  • wpdev_on_load( $callback, $priority = 10 ) — schedule registration on wpdev_load
  • wpdev_on_admin_pages( $callback, $priority = 10 ) — schedule page instantiation on wpdev_admin_pages
  • wpdev_services( $id = null ) — access ajax, form, modal, screen_options, tour, view
  • wpdev_register_service( $id, $service, $replace = true ) — register a custom service
  • wpdev_register_admin_page( $class_name, $args = array(), $priority = 100 ) — rebind a page class under a different menu/context
  • wpdev_register_playground_panel( $module_id, $args = array(), $replace = true ) — add a dev-only playground panel
  • wpdev_dispatch_event( $slug, $payload = array() ) — dispatch a framework/domain event
  • wpdev_register_event_listener( $slug, $callback, $priority = 10, $accepted_args = 1 )
  • wpdev_register_global_event_listener( $callback, $priority = 10, $accepted_args = 2 )

Example — load a module and register runtime work

wpdev_load_module( 'settings-panel-builder' );

wpdev_on_load(
	static function () {
		wpdev_register_event_listener(
			'membership_created',
			static function ( array $payload ) {
				wpdev_log_add( 'Membership created: ' . ( $payload['id'] ?? 'unknown' ) );
			}
		);
	}
);

Example — rebind a production admin page

wpdev_register_admin_page(
	\WPDevFramework\Admin_Pages\Product_List_Admin_Page::class,
	array(
		'parent'     => 'wpdev-playground',
		'context'    => 'admin',
		'capability' => 'wpdev_read_products',
		'module_id'  => 'wpdev-products',
	),
	100
);

Example — register a playground panel

if ( defined( 'WPDEV_PLAYGROUND_DIR' ) ) {
	wpdev_register_playground_panel(
		'my-module',
		array(
			'title'  => __( 'My Module', 'wpdev' ),
			'type'   => 'render',
			'render' => static function () {
				echo '<div class="wrap"><h1>My Module</h1></div>';
			},
		)
	);
}

Note: register entities on wpdev_load; instantiate page classes on wpdev_admin_pages. Services and registries are rebuilt on each request. Do not bypass Module_Loader or add runtime PHP back under inc/.

02

field-builder

Dependencies: core · UI

← back to grid

Field type registry and view resolution used by settings, admin pages, metaboxes, forms, and checkout fields.

Key APIs

  • wpdev_register_field_type( $type, $config = array(), $replace = true )
  • wpdev_get_field_type( $type )
  • wpdev_has_field_type( $type )
  • wpdev_list_field_types()
  • wpdev_unregister_field_type( $type )
  • wpdev_field_view_context_map() — return context-to-view-root map
  • wpdev_field_view( $context, $type ) — resolve a template path for a field type
  • wpdev_checkout_field_view( $type ) — shortcut for checkout field templates

Example — register a custom field type

wpdev_on_load(
	static function () {
		wpdev_register_field_type(
			'my_slug',
			array(
				'sanitize' => 'sanitize_key',
				'view'     => 'admin-pages/fields/text',
			)
		);
	}
);

Example — resolve a field template

$template = wpdev_field_view( 'admin', 'text' );

wpdev_get_template(
	$template,
	array(
		'id'    => 'my_field',
		'name'  => 'my_field',
		'value' => '',
	)
);

Example — customize sanitization

add_filter(
	'wpdev_field_validate_my_slug',
	static function ( $value ) {
		return sanitize_key( $value );
	}
);

The registry does not save field values; the caller saves values through settings, forms, or model handlers. Capability checks belong to the consuming page/form, not the field registry. Prefer wpdev_field_view() over hardcoded view paths.

03

form-builder

Dependencies: core, field-builder · UI

← back to grid

Registers and renders modal or inline forms. Wraps legacy form behavior behind service-aware helpers and integrates with the core ajax/modal services.

Key APIs

  • wpdev_form_api() — return active form service/manager
  • wpdev_register_form( $form_id, $atts = array() ) — register a form definition
  • wpdev_get_form_url( $form_id, $atts = array(), $inline = false ) — build modal/inline URL for a registered form
  • wpdev_modal_open( $form_id, $args = array() ) — return modal opener markup
  • wpdev_register_ajax_modal( $id, $callback, $args = array() ) — register ajax modal content
  • wpdev_ajax_modal_url( $id, $args = array() ) — build ajax modal URL
  • wpdev_render_entity_list_modal( $entities, $args = array() ) — render entity picker modal
  • add_wubox() — enqueue Wubox assets

Example — register a modal form

add_action(
	'wpdev_register_forms',
	static function () {
		wpdev_register_form(
			'archive_product',
			array(
				'title'      => __( 'Archive product', 'wpdev' ),
				'capability' => 'wpdev_edit_products',
				'fields'     => array(
					'product_id' => array(
						'type' => 'hidden',
					),
					'confirm'    => array(
						'type'  => 'toggle',
						'title' => __( 'I understand this action.', 'wpdev' ),
					),
				),
				'handler'    => 'my_archive_product_handler',
			)
		);
	}
);

Example — open the form from a button

$url = wpdev_get_form_url(
	'archive_product',
	array(
		'product_id' => absint( $product_id ),
	)
);

printf(
	'<a class="button wubox" href="%s">%s</a>',
	esc_url( $url ),
	esc_html__( 'Archive', 'wpdev' )
);

Verify nonces and capabilities in form handlers. Unknown form ids return empty URLs. Use core ajax response helpers for save handlers.

if ( wpdev_get_setting( 'enabled', false ) ) {

	static function () {
		delete_transient( 'my_integration_status' );
	}
);

Per-field capability can be set in field config. Save sanitization flows through the settings save pipeline and field sanitization rules. Production page rendering comes from admin-setting-page.

tab-navigation

Dependencies: core · UI

← back to grid

Renders shared tab UI for list views, settings sections, wizard-like flows, and playground panels.

Key APIs

  • wpdev_render_tab_navigation( $tabs, $options = array() )
  • wpdev_list_table_views_as_tabs( $views ) — convert list-table views to tabs

Example — render manual tabs

$active = sanitize_key( wpdev_request( 'tab', 'general' ) );

wpdev_render_tab_navigation(
	array(
		array(
			'id'      => 'general',
			'title'   => __( 'General', 'wpdev' ),
			'url'     => add_query_arg( 'tab', 'general' ),
			'current' => 'general' === $active,
		),
		array(
			'id'      => 'advanced',
			'title'   => __( 'Advanced', 'wpdev' ),
			'url'     => add_query_arg( 'tab', 'advanced' ),
			'current' => 'advanced' === $active,
		),
	),
	array(
		'id' => 'my-module-tabs',
	)
);

Stateless module — callers own active tab state. Apply capability checks before adding protected tabs to the array. Prefer this renderer over bespoke tab markup.

table-builder

Dependencies: core, tab-navigation · Data

← back to grid

Registers list-table classes and shared list-table helpers. Intentionally lightweight and depends only on core and tab-navigation.

Key APIs

  • wpdev_register_list_table( $table_id, $class_name, $config = null, $replace = true )
  • wpdev_get_list_table( $table_id )
  • wpdev_has_list_table( $table_id )
  • wpdev_list_list_tables()
  • wpdev_unregister_list_table( $table_id )
  • wpdev_render_table_builder_fixture_list_table( $module_id )
  • wpdev_render_playground_list_table( $module_id )

Example — register a list table

wpdev_on_load(
	static function () {
		wpdev_register_list_table(
			'my_products',
			\WPDevFramework\List_Tables\Product_List_Table::class
		);
	}
);

Example — minimal list table class

class My_Product_List_Table extends \WPDevFramework\List_Tables\Base_List_Table {
	public function get_columns() {
		return array(
			'name'   => __( 'Name', 'wpdev' ),
			'status' => __( 'Status', 'wpdev' ),
		);
	}

	protected function column_name( $item ) {
		return esc_html( $item->get_name() );
	}

	protected function user_can_ajax_refresh() {
		return current_user_can( 'wpdev_read_products' );
	}
}

Example — hook the bulk pipeline

add_action(
	'wpdev_bulk_pipeline_before_process',
	static function ( array $context ) {
		if ( ! current_user_can( 'manage_network' ) ) {
			wp_die( esc_html__( 'Not allowed.', 'wpdev' ) );
		}
	}
);

wpdev_register_table() is a DB table registration API from core/domain storage, not the list-table UI registry. Override user_can_ajax_refresh() for customer-facing or non-network contexts. Keep heavy queries out of templates; use query classes/managers.

Example — minimal custom page class

class My_Report_Admin_Page extends \WPDevFramework\Admin_Pages\Base_Admin_Page {
	protected $id = 'my-report';
	protected $type = 'submenu';
	protected $parent = 'wpdev';

Lifecycle helpers

06 — Roadmap

Roadmap for developers

How to build a new feature against WPDev without re-introducing monolith coupling.

  1. 01

    Read context.md

    Get the high-level goals, the modules/examples split, the lifecycle, and the “do / do not” rules. The context is the contract.

  2. 02

    Choose the relevant module

    From the 13 framework modules. For an admin CRUD: admin-page-builder + table-builder + metabox-builder + field-builder.

  3. 03

    Read docs/framework/modules/{module}.md

    Module guide: when to use, how to load, important APIs, practical examples, implementation notes.

  4. 04

    Confirm signatures in docs/api/manifest.json

    Never guess. The manifest is generated and the raw API is the source of truth. Read the per-module API_DOC.md too.

  5. 05

    Build under wpdev-examples/{slug}/

    Use the bare folder slug. The module id uses the wpdev- prefix, but the folder name does not. Never wpdev-examples/wpdev-products/ — use wpdev-examples/products/.

  6. 06

    Register entities on wpdev_load

    Use wpdev_on_load() to schedule callbacks. Register tables, settings, forms, fields, widgets, metaboxes, and ajax handlers here.

  7. 07

    Instantiate pages on wpdev_admin_pages

    Use wpdev_on_admin_pages() and wpdev_register_module_admin_pages(). Never instantiate page classes on file load.

  8. 08

    Add playground coverage

    Ship a sandbox panel in wpdev-playground (playground-{module}/playground.php) when WPDEV_PLAYGROUND_DIR is defined. Playground demos prove the API works in isolation.

  9. 09

    Run composer ci

    The CI-safe validation command. Includes smoke, module naming, API docs freshness, view escaping, SQL installer, WaaS example security audit, and PHPUnit (292 tests).

08 — Readiness

Quality gates

What passes today, what is intentionally left for runtime/Docker validation, and the warning categories that do not block CI.

✓

composer ci · passing

The CI-safe validation command. Includes smoke, module naming, framework wording, stale modules/wpdev-* paths, example API usage, playground API docs, API doc schema/order, API docs freshness and top symbols, ajax exception docs, inc/ shim removal, legacy root class shim coverage, sunrise path audit, P3 performance heuristic audit, view escaping check, TODO audit, SQL installer audit, WaaS example security audit (no P0), and PHPUnit (292 tests, 1649 assertions).

!

Runtime regression needs WordPress + MySQL or Docker

manual WordPress multisite verification in the local environment fails on WordPress bootstrap and DB-dependent checks because the configured DB host mysql is not resolvable. This is an environment / connectivity blocker, not a code regression. Re-run with manual multisite smoke test with wpdev + wpdev-examples + wpdev-playground active (or with a healthy local MySQL-backed WordPress multisite) to complete runtime sign-off.

i

Known warning categories (non-blocking)

  • PHPUnit reports 1 warning, 22 deprecations, 1 skipped test. These are not failures today; they should be cleaned before a polished release.
  • WaaS example security audit reports warning-level P1 items: nonce / capability review needed in selected example handlers (site duplication, customer-panel, membership, site, domain, email, broadcast, webhook, event, addon).
  • SQL installer audit: 0 real injection risks; phpcs:ignore lines near $wpdb should still be manually reviewed.
  • wp-config.php in the local environment emits a warning that WPDEV_PLAYGROUND_DIR is already defined.
✓

Docs freshness

The generated manifest is timestamped 2026-06-04. After API changes, refresh with composer docs:api-inventory, composer docs:api-index, and composer docs:api-audit.

Final assessment

The original framework goal has largely been achieved at the code architecture level. Legacy monolith runtime is gone from inc/. Framework primitives are centralized under modules/. WaaS / business domains are optional examples under wpdev-examples (sibling plugin). API docs and a manifest exist for AI and dev consumption. Builder modules expose uniform registry facades. CI-safe validation passes. The remaining gap is runtime release confidence, not basic framework architecture — a healthy MySQL/Docker multisite run is needed to finish the sign-off.
