# Patches & Upgrades

> How to apply a patch to the starter framework, and how to keep a
> consumer's project in sync with newer versions of the starter.

## The patch model

The starter ships as a **git-mergeable** framework, not as a
`vendor/` copy. This is a deliberate choice:

| Model | Pros | Cons |
|---|---|---|
| `vendor/` copy (Composer's `cweagans/composer-patches`) | Truly isolated, no surprise changes | Hard to upgrade, no PR diff to consumer |
| git merge (current) | One `git pull` to upgrade, PR-able | Conflicts possible, must test after merge |

The git-merge model is the right one for a starter because the
overlap between starter and consumer is small (most files are
generated, not authored), and conflicts are obvious.

## What's "framework" vs "consumer"

```
wp-starter-kit/                          ← framework (the starter repo)
├── core/packages/                       ← framework
├── core/php/                            ← FRAMEWORK for plugins/themes that don't override
├── core/components/                     ← consumer (mostly empty in starter, full in scaffolded project)
├── core/styles/                         ← consumer
├── core/assets/                         ← consumer
├── dev/                                 ← framework (developer tools)
├── packages/                            ← framework (CLI tools)
├── tests/                               ← mostly framework, partly consumer (when consumer adds tests)
├── docs/                                ← framework
├── project.config.json                  ← CONSUMER (re-generated by scaffold)
├── composer.json                        ← CONSUMER (with starter tooling declared)
└── package.json                         ← CONSUMER
```

**Rule of thumb:** the consumer can freely edit anything under
`core/components/`, `core/styles/`, `core/assets/`, and
`project.config.json` without conflicting with starter upgrades.
Edits to `core/packages/`, `core/php/`, `dev/`, `packages/`, and
`docs/` are framework changes and need a release of the starter.

## Applying a patch

When a starter release fixes a bug, the upgrade is:

```bash
# In your project (which is a git fork of the starter):
git remote add upstream https://github.com/abolfazl-moeini/wp-plugin-starter-kit.git
git fetch upstream
git merge upstream/main
# Resolve any conflicts (usually in core/components/<slug>/ where you've added new code)
npm install
composer install
npm test
composer test
```

If there are conflicts, they're **always** in `core/components/`,
`core/styles/`, or `project.config.json` — the consumer-only
directories. The framework is structured so it doesn't conflict.

## Releasing a patch to the starter

If you fix a bug in the starter itself (in `core/packages/`,
`core/php/`, `dev/`, etc.):

1. Branch off `main`: `git checkout -b fix/short-slug`.
2. Write a failing test that demonstrates the bug.
3. Write the fix.
4. Update the doc that covers the affected area.
5. `npm test` + `composer test` + `composer lint`.
6. Open a PR to `abolfazl-moeini/wp-plugin-starter-kit`.

After review and merge, the change ships in the next starter
release. Consumer projects pull via `git merge upstream/main`.

## Versioning

The starter follows [Semantic Versioning 2.0.0](https://semver.org/).
A consumer's `composer.json` pins to a major:

```jsonc
"require": {
  "@wpsk/starter": "^0.2.0"  // accepts 0.2.x, 0.3.x; not 1.0
}
```

For the git-merge model, the consumer branches off a tag:

```bash
git clone --branch v0.2.0 https://github.com/abolfazl-moeini/wp-plugin-starter-kit.git my-project
```

The consumer's `CHANGELOG.md` records the starter version it was
forked from, so upgrades are traceable.

## Conflict resolution cookbook

| Conflict location | Resolution |
|---|---|
| `project.config.json` | Consumer's value wins. The starter never changes consumer-only fields (slug, globalName, etc.). |
| `core/components/<slug>/script.js` (consumer-added) | Keep the consumer's version. |
| `core/components/<slug>/script.js` (consumer-renamed starter component) | Manual — ask the consumer what to do. |
| `core/packages/<name>/src/index.js` | Starter's version wins. Consumer shouldn't be editing this. If they have, they've forked the package and need to upstream the change. |
| `composer.json` | Both — starter's `require` and `scripts` blocks are appended to the consumer's, not replaced. |
| `package.json` | Same as composer.json. |
| `tests/phpunit/*` | Consumer's tests win, but starter's may have been added. Run all and merge. |

## When to NOT use this model

If a consumer's project grows large enough that the starter
overhead is < 5% of the codebase, consider forking the starter
into a private repo and pinning to a specific tag. The git-merge
model works well up to ~100 components / 50 PHP files; beyond
that, conflicts become frequent and an internal fork is cheaper.

## Test surface (Phase 4)

`tests/patch/upgrade.test.js` runs the upgrade flow against a
fixture consumer project:

- Clone starter at `v0.1.0`.
- Add a consumer component (`core/components/hello-world/script.js`).
- Merge starter `v0.2.0` (which adds a new package,
  `@wpsk/localize`).
- Verify: consumer's component is intact, new package is bundled,
  tests still pass.

This is a smoke test, not a full upgrade integration. Real
upgrades are best validated by running the test suite after
each merge.
