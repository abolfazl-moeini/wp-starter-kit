# Architecture

> Bird's-eye view of the wp-starter-kit — what lives where, how the
> parts talk, and why it's structured this way.

## Three rings

```
+-------------------------------------------+
|                  PROJECT                  |  ← your code, your config
|  (project.config.json, functions.php,     |     (re-generated by scaffold,
|   core/php/, core/components/,            |      customized by hand)
|   core/packages/, core/assets/)           |
+-------------------------------------------+
|                FRAMEWORK                  |  ← reusable, testable
|  (core/packages/build, rule-engine,       |     (versioned, npm-publishable)
|   dependency-extraction, php-test-tools,  |
|   localization)                           |
+-------------------------------------------+
|                  TOOLING                  |  ← off-the-shelf
|  (esbuild, jest, phpunit, phpcs, phpstan, |     (npm / composer dependencies)
|   rector, babel-plugin-makepot, ...)      |
+-------------------------------------------+
```

The **PROJECT** ring is what you change for a single product.
The **FRAMEWORK** ring is the starter itself — it changes when the
starter is upgraded. The **TOOLING** ring is off-the-shelf and
should rarely be hand-edited.

## The five directories at the top

```
wp-starter-kit/
├── core/                 ← FRAMEWORK + your code (mixed)
│   ├── packages/         ← JS workspaces (@wpsk/*)
│   ├── components/       ← per-component JSX/JS+SCSS
│   ├── assets/           ← images, fonts, vendor JS (verbatim copy)
│   ├── styles/           ← global SCSS (editor, front, ...)
│   └── php/              ← PHP functions/classes
├── dev/                  ← developer scripts (translation, rector, ...)
├── packages/             ← standalone packages (not in core/ — see note)
├── tests/                ← TDD-first; mirrors core/ + packages/
├── docs/                 ← you are here
├── project.config.json   ← THE source of truth for the project
├── composer.json         ← PHP deps + scripts
├── package.json          ← JS deps + scripts
├── phpunit.xml.dist      ← PHPUnit config (mirrors core/php layout)
├── phpcs.xml.dist        ← PHPCS config
└── phpstan.neon.dist     ← PHPStan config
```

**Why two package roots (`core/packages/` AND `packages/`)?**

- `core/packages/` — JS workspaces consumed by the build
  (`@wpsk/starter` reads them via npm workspaces). They become
  the `dist/<slug>-deps.js` bundle.
- `packages/` — standalone packages with their own `package.json`
  and tests, **not** part of the consumer's bundle. Examples:
  `@wpsk/translation` (CLI tool), `@wpsk/create-wp-project`
  (scaffold tool).

Splitting them avoids the trap of "everything is a workspace
package" — translation/scaffold are not consumed at runtime, they
are developer tools.

## The build flow

```
project.config.json
        │
        ▼
┌───────────────────────┐
│ core/packages/build/  │  (esbuild CLIs)
│   esbuild-dependencies│
│   esbuild-components  │
│   esbuild-styles      │
└──────────┬────────────┘
           │
           ▼
       dist/        + assets/        + assets/translations/
```

## The runtime flow

```
Browser GET /?p=1
    ↓
WP renders the page → enqueue `my-project-deps` + `my-component-hello-world`
    ↓
Browser loads dist/my-project-deps.js
    → IIFE that initializes window.MyProject = { hooks, ui, utils, ... }
    → window.MyProjectLoc is populated by wp_localize_script
    ↓
Browser loads dist/components/hello-world.js
    → imports from window.MyProject.ui, .hooks, .utils
    → mounts <HelloWorld /> at <div data-component="hello-world">
    ↓
User interacts → @wpsk/hooks fires → JS-side filter chain → optional fetch
    ↓
`fetch(window.MyProjectLoc.restUrl + 'items', { headers: { 'X-WP-Nonce': ... } })`
    ↓
WP REST handler in core/php/REST/ItemsController.php
    → `apply_filters('my-project/rest/items', $items, $request)`
    → returns JSON
    ↓
JS receives, updates state, re-renders
```

## The PHP↔JS contract (one slide)

| Concept    | JS side                            | PHP side                                                       |
| ---------- | ---------------------------------- | -------------------------------------------------------------- |
| Hooks      | `addAction('my-project/...', cb)`  | `do_action('my-project/...', $data)`                           |
| Filters    | `addFilter('my-project/...', cb)`  | `apply_filters('my-project/...', $data)`                       |
| Data       | `window.MyProjectLoc = {...}`      | `wp_localize_script('my-project-deps', 'MyProjectLoc', $data)` |
| REST       | `fetch(restUrl + 'items')`         | `register_rest_route('my-project/v1', '/items', [...])`        |
| i18n       | `__('Hello', 'my-project')`        | `__('Hello', 'my-project')`                                    |
| Components | `core/components/<slug>/script.js` | `get_template_part('components/<slug>')`                       |
| Styles     | `dist/components/<slug>.css`       | `wp_enqueue_style('my-project-<slug>')`                        |

The hook prefix (`my-project`) is the namespace that ties both
sides together. See `hooks.md`.

## Module boundaries

### `@wpsk/starter` (`core/packages/starter/`)

The "import everything from a single name" entry point:

```js
import { useState, createElement, hooks, ui } from "@wpsk/starter";
```

Re-exports from `@wpsk/hooks`, `@wpsk/ui-components`, `@wpsk/utils`,
`@wpsk/localize`, etc. — so a component only needs to depend on
`@wpsk/starter` to get the full surface.

### `@wpsk/hooks` (`core/packages/hooks/`)

Wraps `@wordpress/hooks` with the project's `hookPrefix` pre-bound.
See `hooks.md`.

### `@wpsk/ui-components` (`core/packages/ui-components/`)

The shared UI primitives: `<Button>`, `<Modal>`, `<Notice>`, etc.
Pure Preact/React (no WP dependencies). The components are
framework-agnostic — see `react-preact-switch.md`.

### `@wpsk/utils` (`core/packages/utils/`)

`deepClone`, `debounce`, `throttle`, `chunk`, etc. The kind of
thing Lodash used to provide, hand-rolled and tree-shake-friendly.

### `@wpsk/localize` (`core/packages/localize/`)

`getLocalize()` — typed accessor for `window[localizeVar]`.
See `localize-contract.md`.

### `@wpsk/rule-engine` (`core/packages/rule-engine/`)

Declarative signal engine. See `signals.md`.

### `@wpsk/translation` (`packages/translation/`)

CLI tool for the translation pipeline. **Not in `core/`.**
See `translation.md`.

### `@wpsk/create-wp-project` (`packages/create-wp-project/`)

Scaffold CLI. **Not in `core/`.** See `scaffold.md`.

### `@wpsk/php-test-tools` (`core/packages/php-test-tools/`)

PHPUnit + PHPCS + PHPStan configs and base classes. See
`php-test-tools.md`.

## Testing strategy

```
core/
├── packages/hooks/         →  tests/packages/hooks.test.js     (50 tests)
├── packages/ui-components/ →  tests/packages/ui-components.test.js (12 tests)
├── packages/rule-engine/   →  tests/packages/rule-engine.test.js (18 tests)
├── packages/utils/         →  tests/packages/utils.test.js
├── packages/localize/      →  tests/packages/localize.test.js
├── packages/build/         →  tests/integration/build.test.js
└── components/<slug>/      →  tests/components/<slug>.test.js
    (per component, optional)

core/php/                   →  tests/phpunit/
                              TranslationPipelineTest, TranslationBootstrapTest,
                              PluginBaseTestCase usage, etc.
```

**Convention:** test files mirror source layout. `core/packages/hooks/src/index.js`
↔ `tests/packages/hooks.test.js`. The mirror is enforced by a CI check
(`composer test:mirror`).

## What goes where (decision tree)

| I want to …                                             | Where it goes                                     |
| ------------------------------------------------------- | ------------------------------------------------- |
| Add a new shared JS module that multiple components use | `core/packages/<name>/src/index.js`               |
| Add a new component                                     | `core/components/<slug>/script.js` + `style.scss` |
| Add a new global SCSS file                              | `core/styles/<name>.scss`                         |
| Add a new PHP function/class                            | `core/php/<name>.php`                             |
| Add a new REST endpoint                                 | `core/php/REST/<Name>Controller.php`              |
| Add a new test                                          | mirror the source path under `tests/`             |
| Add a new doc                                           | `docs/<name>.md`                                  |
| Add a new translation string                            | wrap with `__()` in JS or PHP                     |
| Add a new environment-specific config                   | `dev/<env>/<name>.php` (not `core/php/`)          |
| Add a new dev tool (CLI, generator)                     | `packages/<name>/` (not `core/packages/`)         |

If you're unsure, ask: "Will this be consumed at runtime in
the consumer's bundle?" If yes, `core/`. If no, `packages/` or `dev/`.
